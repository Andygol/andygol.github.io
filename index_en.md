#### Now more detailed.

OSM is 2-dimensional maps. That is, the map is flat without taking into account irregularities of terrain. All those are on the surface, by default, is taken as `layer=0` (surface of rivers – as well), what over the surface – `layer=1` and etc. (bridges, trestle, etc.), what is under the ground(water) – `layer=-1` and etc. (tunnels, culverts, underground structures). But despite all this two-dimensionality, we can operate the third dimension, indicating the height of buildings, trees, fences and barriers; depth of water bodies and so forth. As a result, we get close to the real (without topography) picture of the world around us.

## 1. Retrieving data
 
In order to create a 3D map, we need to get the data, based on which we build our map. For this purpose objects must be specified with data about their height. In the case of specifying height or amount of levels for multi storey buildings the situation in Dnipropetrovsk is not so bad. At least, most of the buildings in the downtown area and in areas high-rise buildings have information about the quantity of floors and / or indication of their height.


To get the data I used the query on the server [overpass-turbo.eu](http://overpass-turbo.eu). Not to write the query manually use the built-in assistant:

![overpass_helper](https://cloud.githubusercontent.com/assets/369696/7304052/531d8924-e9fd-11e4-9006-f6883924f103.png)

As a result we generate a query: 
```xml
/*
This has been generated by the overpass-turbo wizard.
The original search was:
“building in Dnipropetrovsk”
*/
[out:json][timeout:25];
// fetch area “Dnipropetrovsk” to search in
{{geocodeArea:Dnipropetrovsk}}->.searchArea;
// gather results
(
  // query part for: “building”
  way["building"](area.searchArea);
  relation["building"](area.searchArea);
);
// print results
out body;
>;
out skel qt;
```
But let's not hurry up and abort its automatic execution, as we need to add to it few parameters. Add parameters to obtain data on buildings marked `building:part` tag

```xml
way["building:part"](area.searchArea);
relation["building:part"](area.searchArea);
```
and to increase the parameter `[timeout:*]` (waiting time of the query) from 25 to 500 seconds. Now try to execute [our request](http://overpass-turbo.eu/s/8XX) in browser. Just want to warn you if you have less than 8GB RAM your browser will probably not be able to process the data. How to get out of this situation? It's simple – use the «Export».  It will be exported directly into JOSM

![overpass_export](https://cloud.githubusercontent.com/assets/369696/7304095/a0d82084-e9fd-11e4-9f4d-697bac4f48b8.png)

To do this you should open JOSM with enabled "Remote Control".

![overpass_export_fix](https://cloud.githubusercontent.com/assets/369696/7304123/cef965e0-e9fd-11e4-83e6-e94d33a2b97b.png)

Assistant suggests you to correct the request so that the data will be obtained in XML-format, JOSM processes data in this format. Press the "Fix query" and switch to JOSM. After loading the data can be saved to a file (*.osm)

![josm_with_data](https://cloud.githubusercontent.com/assets/369696/7304148/fc5a3974-e9fd-11e4-920c-1b1bcf692c12.png)

## 2. Installing SQL Server

Use [PostgreSQL](http://wiki.openstreetmap.org/wiki/PostgreSQL) with [PostGIS](http://wiki.openstreetmap.org/wiki/PostGIS) to store data in the database.
For instructions on installing the server and initializing the database [here](http://wiki.openstreetmap.org/wiki/PostGIS/Installation). Follow the instructions for your operating system.

[Start Server](http://www.postgresql.org/docs/9.4/static/server-start.html) with a command  
`$ postgres -D /usr/local/pgsql/data`

in my case 
`andygol$ postgres -D /usr/local/var/postgres`

[activate PostGIS](http://postgis.net/docs/postgis_installation.html#install_short_version)  
`andygol$ psql gisdb`
`gisdb=# CREATE EXTENSION postgis;`  
or  
`andygol$ psql -d gisdb -c 'CREATE EXTENSION postgis;'`

## 3. Export data to a database

To export data, use the [osm2pgsql](http://wiki.openstreetmap.org/wiki/Osm2pgsql)

Execute the command: 
`osm2pgsql -s -S ~/Documents/Mapbox/Studio/3dp/dp-default.style -d gisdb -C 4000 --number-processes 2 dp-buildings.osm`

See the option
`-S ~/Documents/Mapbox/Studio/3dp/dp-default.style`

it points to the location of a file with the indication of fields that will be used to import data into the database. Since the standard file does not include all the required fields add them into your file:
```
  node,way   building:part text        polygon #add for 3d
  node.way   building:levels text      polygon #add for 3d
  node,way   building:height text      polygon #add for 3d
  node,way   building:min_level text   polygon #add for 3d
  node,way   height       text         polygon #add for 3d
  node,way   min_height   text         polygon #add for 3d
```
http://wiki.openstreetmap.org/wiki/Osm2pgsql#Import_style

## 6. Creating SQL queries

The main parameter to build our 3D maps is data of the height of buildings, `height` tag. But as it turns out , very small number of buildings have data about their height. However, we have data on the number of floors of some buildings, `building:levels` tag. Let use it:
``` sql
SELECT way, height
FROM planet_osm_polygon
WHERE ((height IS NOT NULL) AND ("building:levels" IS NOT NULL)) 
 /*buildings that have height and number of floors*/
ORDER BY ST_YMax(ST_Envelope(way))
```
```sql
SELECT way, height
FROM planet_osm_polygon
WHERE ((height IS NOT NULL) AND ("building:levels" IS NULL)) 
 /*buildings that have height but no storeys*/
ORDER BY ST_YMax(ST_Envelope(way))
```
```sql
SELECT way, (CAST ("building:levels" AS int) * 3 + 3) as height
FROM planet_osm_polygon
WHERE ((height IS NULL) AND ("building:levels" IS NOT NULL)) 
/*buildings which do not have the height but have a number of floors*/
ORDER BY ST_YMax(ST_Envelope(way))
```
Finally buildings which do not have height data and the number of floors
```sql
SELECT way, 4.5 AS height
FROM planet_osm_polygon
WHERE (height IS NULL AND "building:levels" IS NULL)
ORDER BY ST_YMax(ST_Envelope(way))
```
